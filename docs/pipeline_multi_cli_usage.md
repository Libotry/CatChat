# CatChat 流水线多端口多进程使用说明

本文档说明如何在流水线模式下，为每只猫猫使用独立 CLI 端口，并由用户手动拉起对应数量的 CLI 窗口。

## 1. 目标与机制

在流水线模式中，系统采用以下机制：

- 每只猫猫都有独立端口（pipelineCliPort）。
- 每只猫猫有独立 CLI 进程（可理解为独立 CLI 框）与独立端口（pipelineCliPort）。
- 用户手动为每只猫启动独立 CLI 实例：node cat_chat_cli.js --port <猫端口>。
- 前端启动流水线前，只做端口健康检查，不再自动拉起进程。
- 每只猫的模型切换命令（如 use-opus46）在该猫进程启动时执行并固定在该进程环境中，不再在请求期共享切换。
- 每次流水线调用会携带 catId + catCliPort，后端按端口路由到对应猫猫进程。
- 若端口冲突，后端自动避让到可用端口，并把最终端口回传前端。
- 工作路径输入框中的目录会作为所有猫猫后续任务的默认工作目录，并在提示词中授予该目录及子目录的读/写/执行权限。

Windows 多窗口说明：

- 在 Windows 下，后端会使用 Start-Process 为每只猫拉起独立控制台窗口。
- 每个窗口绑定一只猫的模型环境与端口，可并发执行且互不干扰。

## 2. 启动顺序

### 步骤 1：启动本地 CLI 代理总调度

在仓库根目录执行：

```bash
node cat_chat_cli.js --port 3460
node cat_chat_cli.js --port 3461
node cat_chat_cli.js --port 3462
```

说明：

- 每个窗口启动一个实例，对应一只猫。
- 端口号需要与前端该猫配置一致。

默认监听：

- 代理服务：http://localhost:3456
- 健康检查：GET /health
- Claude 请求：POST /claude-code

### 步骤 2：打开前端页面并连接代理

1. 打开 cat_chat.html。
2. 在左侧本地 CLI 代理区域，确认地址为 http://localhost:3456。
3. 开启代理并测试连接。

### 步骤 3：在流水线面板准备任务

在流水线模式中：

1. 填写需求描述。
2. 填写输出目录（即默认工作目录）。
3. 设置单任务超时。
4. 给每只猫设置模型切换命令（可选）：use-glm / use-opus46 / use-kimi。
5. 给每只猫设置 CLI 端口（或点击“自动分配端口”）。

说明：

- 你在工作路径框中填写目录后，后续流水线对话默认都以该目录为当前工作目录。
- 同步进程时会按猫携带 switchCommand，让每只猫在自己的独立 CLI 进程里固定模型环境。

### 步骤 4：检查猫猫进程（手动模式）

点击“同步猫猫进程”。

- 成功后会显示每只猫的状态与端口（connected/offline）。
- 若某只猫离线，请手动在独立窗口启动该端口实例。
- Windows 下会看到新的独立控制台窗口弹出（每只猫一个）。

### 步骤 5：启动流水线

点击“启动流水线”。

启动时会再次自动同步一次猫猫进程，然后按阶段执行：

- 设计开发
- 代码检视
- 测试报告

## 3. 前端操作说明

### 3.1 每只猫设置项

在成员列表（流水线模式）中，每只猫有两项关键配置：

- 模型切换：use-glm / use-opus46 / use-kimi / 不切换
- CLI 端口：1024-65535

模型切换规则：

- 切换命令在“该猫进程启动时”执行。
- 每只猫独立进程互不影响，支持并发。
- 修改模型切换后，执行一次“同步猫猫进程”即可生效。

### 3.2 自动分配端口

点击“自动分配端口”后，会从 3460 开始顺序分配给当前猫猫。

### 3.3 状态说明

猫猫进程状态可能显示：

- started：本次新拉起
- reused：复用已有进程

若出现端口映射：

- 例如 3460 -> 3463，表示 3460 不可用，后端已自动改为 3463。

## 4. 后端接口说明

### 4.1 GET /health

返回代理健康信息，并包含当前已管理的猫猫进程摘要。

### 4.2 /pipeline/sync 说明

当前为手动多窗口模式，不再使用自动拉起接口。

- 该接口会返回 manual 模式提示。
- 由用户自行拉起各猫猫端口实例。

### 4.3 POST /claude-code

流水线调用时会携带：

- catId
- catCliPort
- prompt
- timeoutMs
- workingDir

后端据此把请求转发到对应猫猫端口的 minimal-claude.js 实例。

补充：

- 每个 /claude-code 请求会发到对应猫端口实例。
- switchCommand 由该猫实例处理，不会在不同猫之间共享干扰。

## 5. 常见问题与排查

### 5.1 点击同步失败

检查：

1. cat_chat_cli.js 是否已启动。
2. 前端代理地址是否正确（默认 http://localhost:3456）。
3. 端口是否被系统策略拦截。

### 5.2 某只猫一直无响应

检查：

1. 状态区该猫是否 connected。
2. 该猫端口实例是否已手动启动：node cat_chat_cli.js --port <该端口>。
3. 模型切换命令是否有效（例如 use-opus46）。
4. 工作路径是否可访问（目录不存在会自动创建；若权限不足请更换路径）。

### 5.3 端口冲突

系统会自动避让。若你希望固定端口：

1. 手动改一个空闲端口。
2. 再点“同步猫猫进程”。

### 5.4 需要彻底重来

推荐顺序：

1. 结束流水线。
2. 点击“重置”。
3. 重新分配端口。
4. 重新同步猫猫进程。
5. 重新启动流水线。

### 5.5 如何关闭多窗口

- 直接关闭对应猫猫控制台窗口。
- 或在窗口内 Ctrl+C 结束 node cat_chat_cli.js 实例。

## 6. 最佳实践

- 先“自动分配端口”，再按需微调。
- 启动流水线前先点一次“同步猫猫进程”（健康检查）。
- 保持每只猫端口唯一。
- 当猫猫数量变化（新增/删除）后，先手动增减 CLI 窗口，再同步再启动。
- 模型切换变更后，建议重启该猫对应端口实例。
- 工作路径建议使用项目目录，避免使用系统受限目录。
